# Selective Disclosure for JSON Web Tokens (SD-JWT) — RFC 9901

## 概要

SD-JWT（Selective Disclosure for JSON Web Tokens）は、JWT ペイロード内の個別クレームに対して選択的開示を実現する IETF 標準仕様です。[RFC 9901](https://www.rfc-editor.org/rfc/rfc9901.html) として 2025年11月19日に Standards Track RFC として発行されました。

発行者（Issuer）はユーザーの全クレームを含む署名付き JWT を保有者（Holder）に発行します。保有者は検証者（Verifier）ごとに必要なクレームのみを選択して開示できます。例えば年齢確認時には生年月日のみ、住所確認時には住所のみを提示でき、他のクレームは秘匿されたままです。

技術的には「塩付きハッシュ」アプローチを採用します。平文クレームを JWT ペイロードに直接格納する代わりに、ハッシュダイジェストのみを署名付き JWT に含め、実際のクレーム値は別途「Disclosure」として管理します。検証者はハッシュを再計算して署名ペイロードのダイジェストと照合することで、開示されたクレームの真正性を確認します。

## 背景と経緯

デジタルアイデンティティシステムでは、クレデンシャルに含まれるクレームの「過剰共有」が長年の課題でした。従来の JWT ベースのクレデンシャルでは、いずれかのクレームを提示するために JWT 全体を渡す必要があり、必要以上の個人情報が検証者に開示されます。

プライバシーに配慮したクレデンシャルシステムとして先行していたのは AnonCreds（Hyperledger Indy の CL 署名）や BBS+ 署名です。これらは強力な匿名性を提供しますが、実装の複雑さから普及が限定的でした。

IETF OAuth Working Group は「既存の JWT エコシステムを活かしながら選択的開示を実現する」という設計方針で SD-JWT を策定しました。2022年の初期ドラフトから約3年の開発期間を経て、eIDAS 2.0 EUDI Wallet や OpenID for Verifiable Presentations (OID4VP) の基盤技術として採用が進む中、2025年11月に RFC として正式発行されました。

## 設計思想

### JWT との互換性優先

SD-JWT の署名済み JWT 部分は標準的な JWS（JSON Web Signature）です。既存の JWT 検証ライブラリで解析でき、新たな暗号ライブラリを導入せずとも基本的な署名検証が可能です。BBS+ や AnonCreds のような高度な暗号スキームが不要なため、開発者の習得コストと実装リスクを最小化しています。

### データ最小化の実用的な実現

完全な匿名性（unlinkability）ではなく「必要最小限のクレームのみを開示する」データ最小化を目標としています。GDPR やプライバシーバイデザインが要求するデータ最小化の原則を、実装しやすい形で提供します。

### トレードオフの明確化

|                  | SD-JWT   | BBS+       | AnonCreds   |
| ---------------- | -------- | ---------- | ----------- |
| 実装の容易さ     | 高       | 中         | 低          |
| 既存 JWT 互換性  | 高       | 低         | なし        |
| Unlinkability    | 限定的   | 署名レベル | 完全        |
| ゼロ知識証明     | 不可     | 限定的     | 可          |
| 標準化ステータス | RFC 9901 | 草案段階   | Hyperledger |

SD-JWT は協調する発行者と検証者が同一のクレデンシャルを識別できます。バッチ発行（異なる鍵・ランダム化されたタイムスタンプを持つ複数クレデンシャルの発行）によって検証者間のリンク可能性を軽減できますが、完全な匿名性は提供しません。

## 技術詳細

### 基本構造

SD-JWT の完全な形式（SD-JWT+KB）は次のようなチルダ（`~`）区切りの文字列です。

```
<Issuer-signed JWT>~<Disclosure 1>~<Disclosure 2>~...~<KB-JWT>
```

各要素は Base64url エンコードされています。Key Binding が不要な場合は KB-JWT を省略できます。

### Issuer-signed JWT

発行者の秘密鍵で署名された JWS です。ペイロードには通常のクレーム（`iss`、`sub`、`exp` など）に加え、選択的開示クレームのハッシュダイジェストを含む `_sd` 配列と、使用するハッシュアルゴリズムを示す `_sd_alg` が含まれます。

```json
{
  "iss": "https://issuer.example.com",
  "iat": 1694900000,
  "exp": 1698500000,
  "_sd_alg": "sha-256",
  "_sd": [
    "j35wlGQlyQ8b4OE3Py6l3AAvOskjcNOxj0SsiVSrVdm",
    "k78zxHRmNvQ6cP9F5jK2LBnwTuLqR3YtVwXyZaBpQen",
    "m91aBcDeFgHiJkLmNoPqRsTuVwXyZ2a3bCdEfGhIjK0"
  ],
  "cnf": {
    "jwk": { "kty": "EC", "crv": "P-256", "x": "...", "y": "..." }
  }
}
```

### Disclosures（開示値）

各 Disclosure は JSON 配列をBase64url エンコードしたものです。

**オブジェクトプロパティの場合（3要素）:**

```
["<salt>", "<claim_name>", <claim_value>]
```

```
["eJ2xLOe8qsiNe0H5CHW79A", "given_name", "太郎"]
```

**配列要素の場合（2要素、claim_name なし）:**

```
["<salt>", <element_value>]
```

ソルト（`salt`）はクレームごとに一意の暗号乱数で、最低128ビットのエントロピーが必要です。

### ハッシュの計算

検証者は次の手順で Disclosure を検証します。

1. Disclosure の文字列（Base64url エンコード形式）を UTF-8 バイト列として取得
2. `_sd_alg` で指定されたアルゴリズム（通常 SHA-256）でハッシュを計算
3. 計算したハッシュが `_sd` 配列内に存在することを確認

この検証により、保有者がクレーム値を恣意的に改ざんできないことが保証されます。

### ネストされた選択的開示

`address` クレームのようなオブジェクト型のクレームでは、階層的な選択的開示が可能です。

```json
{
  "_sd": ["<address_hash>"]
}
```

`address` を開示すると、その値として次のようなオブジェクトが現れます。

```json
{
  "_sd": ["<street_hash>", "<city_hash>", "<country_hash>"]
}
```

これにより、`address` を開示しつつも `street_name` のみを秘匿するといった細粒度の制御が可能です。

### Key Binding JWT（KB-JWT）

KB-JWT は保有者が対応する秘密鍵を保有していることを証明します。クレデンシャル流出後の無権限使用を防ぐために重要です。

ヘッダには `"typ": "kb+jwt"` を指定し、ペイロードには以下の必須クレームを含めます。

| クレーム  | 説明                                       |
| --------- | ------------------------------------------ |
| `iat`     | 署名生成時刻                               |
| `aud`     | 検証者の識別子（リプレイ攻撃防止）         |
| `nonce`   | トランザクション固有のランダム値           |
| `sd_hash` | 提示された SD-JWT（KB-JWT 除く）のハッシュ |

検証者は `aud` と `nonce` を検証することで、別のトランザクションや検証者への転用（リプレイ攻撃）を検出できます。

### JSON シリアライゼーション

コンパクト形式に加え、JWS JSON Serialization も使用できます。`disclosures` と `kb_jwt` をアンプロテクテッドヘッダパラメータとして格納し、JSON ベースの API やメッセージングで扱いやすい形式を提供します。

## 実装上の注意点

### ハッシュの照合を省略してはならない

最も重要なセキュリティ要件です。RFC 9901 は「Disclosure のハッシュを確認せずに全クレームを抽出する単純な実装は攻撃に脆弱である」と明示しています。検証者は必ず Disclosure のハッシュを計算し、`_sd` 配列内の値と一致することを確認しなければなりません。

### ソルトのエントロピー

各 Disclosure のソルトは最低 128 ビットの暗号乱数を使用してください。エントロピーが不足すると、辞書攻撃やプリイメージ攻撃により秘匿クレームの値を推測される可能性があります。

### クレーム名の衝突

選択的開示クレームの名前が、JWT ペイロードの平文クレームと重複することを防いでください。RFC 9901 は衝突を検出した場合に SD-JWT 検証を失敗させることを要求しています。

### `none` アルゴリズムの禁止

Issuer-signed JWT への署名は必須です。`none` アルゴリズムを使用することは厳禁です。

### リンク可能性の限界

協調する発行者と検証者は、同一のクレデンシャルを異なる提示シーンで同定できます。高いプライバシー要件がある場合は、バッチ発行（同一ユーザーに対して異なる鍵と異なるタイムスタンプを持つ複数クレデンシャルを発行）を検討してください。

### デコイダイジェスト

`_sd` 配列にはダミーのダイジェスト（decoy digests）を含めることができます。これにより、実際の選択的開示クレームの数を秘匿し、どのクレームが開示されていないかの推測を困難にします。

## 採用事例

### EUDI Wallet / eIDAS 2.0

欧州委員会が推進する EU デジタルアイデンティティウォレット（EUDI Wallet）の Architecture Reference Framework (ARF) は、SD-JWT VC を主要なクレデンシャルフォーマットとして採用しています。eIDAS 2.0 規制が求めるデータ最小化原則を SD-JWT が技術的に支える構造です。

### OpenID for Verifiable Presentations (OID4VP)

OID4VP は SD-JWT VC フォーマットのクレデンシャルのプレゼンテーションをサポートします。[OpenID4VC High Assurance Interoperability Profile 1.0](https://openid.net/specs/openid4vc-high-assurance-interoperability-profile-1_0-final.html) では SD-JWT VC が必須フォーマットとして規定されており、政府・金融分野での高保証ユースケースへの採用が進んでいます。

### 実装ライブラリ

複数のオープンソース実装が存在し、主要言語でカバーされています。

- **Python:** `sd-jwt` パッケージ（Authlete、others）
- **TypeScript / JavaScript:** `@sd-jwt/core`（OpenWallet Foundation）
- **Rust:** `sd-jwt-rs`

## 関連仕様・後継仕様

| 仕様                                   | 関係                                                                    |
| -------------------------------------- | ----------------------------------------------------------------------- |
| RFC 7519 (JWT)                         | 依存仕様。SD-JWT は JWS ベースで構築                                    |
| RFC 7515 (JWS)                         | 依存仕様。署名と検証の基盤                                              |
| RFC 7800                               | Key Binding の Proof-of-Possession セマンティクス                       |
| SD-JWT VC (draft-ietf-oauth-sd-jwt-vc) | SD-JWT を用いた Verifiable Credentials の仕様。`vct` クレームなどを追加 |
| OID4VP                                 | SD-JWT VC の提示プロトコル                                              |

SD-JWT VC は SD-JWT を基盤としてクレデンシャル固有の要件を追加しています。`vct`（verifiable credential type）クレーム、発行者・保有者・検証者モデルの検証ルール、表示・検証用メタデータ構造などを規定します。SD-JWT VC 自体は 2026年4月時点でドラフト段階（draft-ietf-oauth-sd-jwt-vc-15）です。
