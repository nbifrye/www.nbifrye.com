# ISO/IEC 18013-5 — Mobile Driving Licence (mDL)

> **発行**: ISO/IEC / **2021年9月30日** / **ステータス**: International Standard (Published)
> **正式タイトル**: Personal identification — ISO-compliant driving licence — Part 5: Mobile driving licence (mDL) application
> **関連規格**: ISO/IEC TS 18013-7:2025（オンライン提示フロー）

---

## 概要

ISO/IEC 18013-5 は、スマートフォン上に格納されたデジタル運転免許証（mDL: mobile Driver's Licence）のデータ形式・発行・提示・検証のプロトコルを定義する国際標準である。物理的な運転免許証の代替としてだけでなく、年齢確認や本人確認のユースケースをカバーする汎用的な身分証明基盤として設計されている。

CBOR（Concise Binary Object Representation）を採用したコンパクトなバイナリ形式（mdoc）、近接・オンライン・オフラインの多様な提示フロー、そして暗号署名に基づく真正性検証が三本柱となっている。米国・EU・日本など世界各国で採用が進み、OpenID4VP や Digital Credentials API との連携により、Web・モバイル双方のデジタルID基盤の中核標準として位置づけられている。

---

## 背景：なぜこの仕様が必要だったか

物理的な運転免許証は磁気ストライプや IC チップを持つ場合があるが、統一されたデジタル提示プロトコルが存在しなかった。各国・各州がバラバラな実装を進めていた状況を受け、ISO/IEC JTC 1/SC 17（Cards and security devices for personal identification）が標準化を主導した。

設計上の主な課題は以下の通りだった：

- **プライバシー保護**: 検証者が必要最小限の属性だけを取得できる選択的開示の実現
- **オフライン動作**: ネットワーク接続なしでの検証（空港・警察官による路上確認など）
- **相互運用性**: 異なる発行者・検証者間での統一フォーマット
- **セキュリティ**: クローニング・改ざん・リプレイ攻撃への対策

従来の X.509 / JSON / XML ベースではなく CBOR を採用したのは、IoT・組み込みデバイスを含む多様な環境でのパフォーマンスと帯域効率を優先したためである。

## 基本概念

### ロール

| ロール          | 説明                                                                                                     |
| --------------- | -------------------------------------------------------------------------------------------------------- |
| **mdoc Issuer** | mDL を発行する機関（運転免許センター・DMV 等）。IACA（Issuer Authority Certificate Authority）として機能 |
| **mDL Holder**  | 免許証保有者。スマートフォン等のウォレットアプリに mDL を格納                                            |
| **mdoc Reader** | mDL を検証する機関・システム（空港・警察・レンタカー会社等）                                             |

### mdoc（Mobile Document）

mdoc は mDL のデータを格納するコンテナ形式であり、CBOR でエンコードされる。`docType` でドキュメント種別を識別し、`org.iso.18013.5.1` が mDL の標準名前空間である。

```
mdoc
├── docType        : "org.iso.18013.5.1.mDL"
├── issuerSigned
│   ├── nameSpaces : 属性データ（選択的開示のためのダイジェスト付き）
│   └── issuerAuth : 発行者の署名（COSE_Sign1）
└── deviceSigned
    ├── nameSpaces : デバイス署名対象の追加属性
    └── deviceAuth : デバイスによる認証（COSE_Sign1 または COSE_Mac0）
```

### IssuerSignedItem

各属性（data element）は以下の構造を持ち、選択的開示を可能にする：

```
IssuerSignedItem
├── digestID      : 整数識別子（MSO の digestIDs に対応）
├── random        : ランダム塩値（ハッシュのプリイメージ攻撃防止）
├── elementIdentifier : 属性名（例："family_name", "birth_date"）
└── elementValue  : 属性値
```

### MSO（Mobile Security Object）

発行者が署名する対象は MSO であり、各 IssuerSignedItem のハッシュダイジェストを含む。検証者は MSO の署名を確認した上で、受け取った属性のダイジストが MSO に記録されたものと一致することを確認する。これにより、属性の選択的開示と改ざん検出が両立される。

## プレゼンテーションフロー

### 1. Proximity（近接提示）

BLE/NFC/QR コードを用いた物理的近接での提示フロー。警察官の路上確認・空港検査場・レンタカーカウンターなどのユースケースが対象。

```
[mDL Holder] ←BLE/NFC/QR→ [mdoc Reader]

1. Reader が QR コードまたは NFC で engagement を開始
2. セッション鍵交換（ECDH による共有秘密導出）
3. Reader が DeviceRequest（要求属性リスト）を送信
4. Holder が DeviceResponse（mdoc）を返送
5. Reader が IssuerAuth および DeviceAuth を検証
```

セッションは暗号化され、中間者攻撃に対する保護が施される。NFC は ISO/IEC 14443 準拠、BLE は GATT プロファイルを使用。

### 2. Online（オンライン提示）

ISO/IEC TS 18013-7:2025 で定義されるリモート提示フロー。OpenID4VP プロトコルを基盤として採用しており、Web ブラウザや API 経由でのデジタルサービス向けの本人確認に対応する。

```
[RP/Verifier] ←HTTP/HTTPS→ [Wallet App]

1. Verifier が OpenID4VP Authorization Request を送信
2. Wallet が対応する mDL を選択し、ユーザー同意を取得
3. DeviceResponse（mdoc）を OpenID4VP Response に格納して送信
4. Verifier がレスポンスを検証
```

Digital Credentials API（W3C）と組み合わせることで、ブラウザレベルでのメディエーションが可能になる。

### 3. Offline

事前にキャッシュされた Issuer Signed データを使用し、オンライン検証なしで提示する方式。有効期間内であれば証明書失効チェック（OCSP/CRL）なしで検証できる。災害時・通信圏外でのユースケースに対応。

## セキュリティ上の重要な考慮事項

### IssuerAuth（発行者認証）

MSO に対する COSE_Sign1 署名。発行者は IACA（Issuer Authority CA）が発行した Document Signer 証明書を使用する。検証者は証明書チェーンを IACA ルートまで検証する。

検証フロー：

1. Document Signer 証明書の有効性確認（有効期間・失効確認）
2. IACA ルート証明書への信頼チェーン確認
3. MSO 署名（COSE_Sign1）の検証
4. 属性ダイジェストの一致確認

### DeviceAuth（デバイス認証）

mDL がウォレットアプリ（デバイス）に結びついていることを証明するメカニズム。以下の2方式がある：

- **DeviceSignature**: デバイス秘密鍵による COSE_Sign1 署名（セッション転記防止のため SessionTranscript を含む）
- **DeviceMac**: ECDH セッション鍵を用いた COSE_Mac0（近接フローで使用）

DeviceAuth のトランスクリプト束縛により、録画した提示フローの再利用（リプレイ攻撃）が防止される。

### 選択的開示

検証者は `ItemsRequest` で必要な属性のみを要求する。Holder はその要求に対応する IssuerSignedItem だけを提示できる（要求されていない属性の IssuerSignedItem は含めない）。検証者は開示された属性のダイジェストのみを確認でき、未開示属性のダイジェストは MSO に記録されているが値は確認できない。

**注意**: ISO 18013-5 の選択的開示は、SD-JWT の方式（ハッシュリスト）と実装レベルでは異なるが、設計思想（属性単位の開示制御）は共通している。

### セキュリティ上の既知リスク

- **証明書管理**: IACA 証明書の管理・失効が各国の行政機関に依存する。IACA ルートの相互認証は現時点では未整備
- **デバイス侵害**: デバイス秘密鍵が漏洩した場合、偽の DeviceAuth が生成される可能性
- **有効期間管理**: MSO の有効期間（`validUntil`）が過ぎた mDL は検証失敗となるが、オフライン環境では失効チェックができない

## 実装状況・採用

### 米国

最も先進的な採用状況を示す。2025年時点で以下の州が mDL を運用中または導入済み：

- **カリフォルニア・アリゾナ・ルイジアナ・メリーランド・コロラド・ジョージア** など18州以上
- TSA は空港セキュリティ（CAT-2 リーダー）で250カ所以上の検査レーンに対応
- Apple Wallet・Google Wallet の両プラットフォームが ISO 18013-5 準拠の mDL をサポート

### EU

eIDAS 2.0 規則により、2026年12月までに全加盟国が EUDI Wallet を市民に提供する義務を負う。EUDI Wallet の必須フォーマットとして mdoc が指定されており、ISO 18013-5 / 18013-7 は事実上の法的義務標準となっている。

### 日本

マイナンバーカードのデジタル化（次世代マイナカード）において mDL 規格との整合が検討されている。2028年以降のスマートフォン搭載型マイナカードでは ISO 18013-5 互換フォーマットの採用が視野に入っている。

### プラットフォーム対応

| プラットフォーム | 対応状況                                                                                  |
| ---------------- | ----------------------------------------------------------------------------------------- |
| **Apple iOS**    | Wallet アプリで mDL 対応（米国各州）、NFC/QR プレゼンテーション                           |
| **Android**      | Google Wallet で mDL 対応、Identity Credential API（Android 11+）でシステムレベルサポート |
| **W3C DC API**   | Chrome/Safari で実装進行中。mDL の Web 提示を標準化                                       |

## 読み解きのポイント

### CBOR vs JSON の設計判断

mdoc が CBOR を採用したのは、バイナリエンコードによるサイズ削減（JSON 比20〜50%削減）とパース速度の優位性のためである。ただしデバッグ・開発体験の観点では JSON より難しく、エコシステム成熟には時間を要した。SD-JWT VC（JSON/JWT ベース）と mdoc（CBOR ベース）の二つのフォーマットが並立している現状は、この設計選択の帰結でもある。

### 18013-5 と 18013-7 の役割分担

ISO 18013-5 はあくまで**近接フロー**の標準である。Web サービスでの本人確認（「ブラウザで mDL を提示してログイン」）は **ISO/IEC TS 18013-7** の領域であり、その実装は OpenID4VP に委ねられている。この分離設計は合理的だが、実装者が把握すべき仕様が複数にまたがる点は複雑さを増す。

### 選択的開示とゼロ知識証明

現行の ISO 18013-5 の選択的開示は属性単位の開示制御であり、「21歳以上であることのみを証明（年齢値は非開示）」といったゼロ知識証明的な開示は標準仕様には含まれない。年齢比較（age_over_NN 要素）は規定されているが、これは「18歳以上か否か」という真偽値の開示であり、厳密な ZKP とは異なる。より高度なプライバシー保護には、BBS+ 署名などを組み合わせた拡張が必要となる。

### IACA 信頼基盤の未解決問題

各国の IACA ルート証明書をどう相互認証するかは、国際的に未解決の課題である。AAMVA（北米）・EU Trusted List・各国政府 PKI が個別に存在しており、グローバルな信頼基盤は現時点では整備されていない。実装では、検証者ごとに信頼する IACA ルートのリストを管理する必要がある。
