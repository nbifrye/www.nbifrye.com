# JWT-Secured Authorization Request (JAR) — RFC 9101

## 概要

JWT-Secured Authorization Request (JAR) は、OAuth 2.0 の認可リクエストパラメータを JWT として署名・暗号化することで、セキュリティを大幅に強化する仕様です。

標準の OAuth 2.0 では、認可リクエストは URL のクエリパラメータとして送信されます。この方式には次の本質的な問題があります。

- **改ざんリスク**: URL パラメータはブラウザのリダイレクト経路（フロントチャネル）を通過するため、中間者による改ざんが可能
- **送信元未認証**: リクエストがどのクライアントから来たものか、暗号的に証明できない
- **機密性の欠如**: `scope` や `claims` など機密パラメータが URL に平文で露出する

JAR はこれらをまとめて解決します。認可パラメータを JWS（JSON Web Signature）で署名し、必要に応じて JWE（JSON Web Encryption）で暗号化した JWT として送信することで、整合性保護・送信元認証・機密性を確保します。

## 背景と経緯

JAR の原型は OpenID Connect Core 1.0 の `request` / `request_uri` パラメータです。OIDC では JWT 形式のリクエストオブジェクトをオプション機能として定義していましたが、OAuth 2.0 単体では対応する仕様が存在しませんでした。

金融 API の標準化が進む中で、FAPI（Financial-grade API）ワーキンググループは OAuth 2.0 の認可リクエストにも同等のセキュリティが必要と判断しました。FAPI 1.0 では JAR の前身となる「Request Object」を必須要件として採用し、その後 OAuth 2.0 のベースドキュメントとして RFC 9101 に昇格しました。

RFC 9101 は OIDC Core の当該機能を OAuth 2.0 に一般化し直した仕様であり、OIDC 依存の排除と OAuth エコシステム全体への展開を意図しています。

## 設計思想

### クエリパラメータからの脱却

標準 OAuth 2.0 の認可リクエストは `response_type`・`client_id`・`redirect_uri`・`scope` などをクエリパラメータとして送信します。これらは HTTP GET の URL に含まれるため、リダイレクト経由で転送されるたびにブラウザの履歴・ログ・中継プロキシに記録される可能性があります。

JAR は「JWT のクレームセット = 認可パラメータ」という設計で、この問題を根本から解決します。JWT はクライアントが秘密鍵で署名するため、パラメータの改ざんはいかなる中継者も不可能です（署名を無効化してしまうため）。

### 2 つの送信方式

JAR には「値渡し（by value）」と「参照渡し（by reference）」の 2 通りの送信方式があります。

| 方式                     | パラメータ    | 動作                                                  |
| ------------------------ | ------------- | ----------------------------------------------------- |
| **Request by Value**     | `request`     | 署名済み JWT 文字列をそのまま認可エンドポイントに送信 |
| **Request by Reference** | `request_uri` | JWT を配置した URL を送信し、サーバーが取得           |

**Request by Value** はシンプルで外部サーバー不要ですが、JWT が長くなると URL が肥大化します。**Request by Reference** は URL を短く保てますが、認可サーバーがリクエスト処理前に URI から JWT を取得するため、DDoS リスクへの対策が必要です（URI は 512 ASCII 文字以内に収めることを RFC は推奨しています）。

### Pushed Authorization Requests (PAR) との補完関係

RFC 9126 の PAR と組み合わせることで、さらに強固な構成が実現します。PAR では認可リクエストをバックチャネル（クライアントから認可サーバーへの直接 HTTPS POST）で事前送信するため、Request Object がフロントチャネルに一切露出しません。

```
Client ──[1: JWT Request Object]──► PAR Endpoint (HTTPS POST)
                                     ◄── request_uri
Client ──[2: request_uri のみ]──► Authorization Endpoint (User Agent)
```

JAR が「何を保護するか」（パラメータの署名・暗号化）を担い、PAR が「どこを通るか」（バックチャネル送信）を担う、明確な役割分担です。

## 技術詳細

### リクエストオブジェクトの構造

リクエストオブジェクトは JWS 署名済み（必要に応じて JWE 暗号化済み）の JWT です。ペイロードのクレームセットには OAuth 2.0 の認可リクエストパラメータを JSON クレームとして格納します。

```json
{
  "alg": "RS256",
  "typ": "oauth-authz-req+jwt"
}
```

```json
{
  "iss": "s6BhdRkqt3",
  "aud": "https://as.example.com",
  "iat": 1311281970,
  "exp": 1311282030,
  "response_type": "code",
  "client_id": "s6BhdRkqt3",
  "redirect_uri": "https://client.example.com/callback",
  "scope": "openid profile email",
  "state": "af0ifjsldkj",
  "nonce": "n-0S6_WzA2Mj",
  "claims": {
    "userinfo": {
      "email": { "essential": true },
      "name": null
    }
  }
}
```

**推奨クレーム:**

| クレーム | 説明                                 |
| -------- | ------------------------------------ |
| `iss`    | 発行者（通常 `client_id` と同値）    |
| `aud`    | 対象認可サーバーのエンドポイント URI |
| `iat`    | 発行時刻（Unix タイムスタンプ）      |
| `exp`    | 有効期限（通常 5〜10 分後）          |

**禁止クレーム:** リクエストオブジェクト内に `request` または `request_uri` を含めてはなりません（[RFC 9101 Section 5](https://www.rfc-editor.org/rfc/rfc9101#section-5)）。

### 認可エンドポイントへの送信

#### Request by Value

```
GET /authorize?
  response_type=code
  &client_id=s6BhdRkqt3
  &request=eyJhbGciOiJSUzI1NiJ9.eyJyZXNwb25zZV90eXBlIjoiY29kZSIs... HTTP/1.1
```

クエリパラメータに `request` を含めます。`client_id` はリクエストオブジェクト外にも送信します（認可サーバーが鍵を取得するために必要）。

#### Request by Reference

```
GET /authorize?
  response_type=code
  &client_id=s6BhdRkqt3
  &request_uri=https%3A%2F%2Fclient.example.com%2Freq%2Fqucsr96342 HTTP/1.1
```

認可サーバーは `request_uri` にアクセスして JWT を取得し、署名を検証してからパラメータを抽出します。

### サーバー側の検証処理

認可サーバーが行うべき検証ステップ（[RFC 9101 Section 7](https://www.rfc-editor.org/rfc/rfc9101#section-7)）:

1. JWS 署名の検証（クライアント登録済み公開鍵を使用）
2. `alg` ヘッダーが許可アルゴリズムリストに含まれることの確認
3. `exp` クレームの有効期限確認
4. `iss` クレームが `client_id` と一致することの確認
5. `aud` クレームが認可サーバー自身のエンドポイント URI と一致することの確認
6. リクエストオブジェクト内の `client_id` とクエリパラメータの `client_id` が一致することの確認

### require_signed_request_object メタデータ

クライアント登録またはサーバーメタデータで `require_signed_request_object: true` を設定すると、JAR が必須化されます。これにより、署名なし通常リクエストへのダウングレード攻撃を防止できます（[RFC 9101 Section 8](https://www.rfc-editor.org/rfc/rfc9101#section-8)）。

### メディアタイプ

RFC 9101 はリクエストオブジェクト用のメディアタイプ `application/oauth-authz-req+jwt` を IANA に登録しています。`typ` ヘッダーにこの値を設定することで、JWT の用途を明示的に宣言できます（Cross-JWT Confusion 攻撃の緩和）。

## 実装上の注意点

### 署名検証の省略厳禁

最も危険な実装ミスは署名検証の省略または `alg: "none"` の許可です。JWS の署名検証はクライアントの事前登録済み公開鍵を使って行い、認可サーバーが許可するアルゴリズムリストを明示的に管理してください。対称鍵（`client_secret`）を JWS 署名に使用することは避けるべきです。

### パラメータマージの罠

OIDC Core では JWT 外のクエリパラメータと JWT 内クレームをマージする実装もありましたが、JAR では **JWT 内のクレームのみが有効** です（[RFC 9101 Section 6.1](https://www.rfc-editor.org/rfc/rfc9101#section-6.1)）。クエリパラメータ層での攻撃を防ぐための仕様上の決定であり、既存の OIDC 実装からの移行時に注意が必要です。

### request_uri 取得の安全対策

`request_uri` を処理する認可サーバーは以下の対策が必要です。

- **タイムアウト設定**: 5〜10 秒以内に取得できない場合はエラー
- **Content-Length チェック**: 異常に大きいペイロードの拒否
- **URI ホワイトリスト**: クライアント登録時に許可 URI を事前登録
- **再帰防止**: 取得した JWT 内に `request_uri` が含まれる場合はエラー（無限ループ防止）

これらを怠ると、悪意のある `request_uri` を使った DDoS 攻撃や Recursive Lookup 攻撃の踏み台になります（[RFC 9101 Section 9.7](https://www.rfc-editor.org/rfc/rfc9101#section-9.7)）。

### Replay 攻撃対策

リクエストオブジェクトには必ず `exp` クレームを設定し、有効期限は 5〜10 分程度に絞ってください。有効期限なし・または非常に長い期限のリクエストオブジェクトは Replay 攻撃のリスクになります。追加対策として、処理済み JWT の `jti` (JWT ID) をキャッシュして重複リクエストを拒否する方法もあります。

### 暗号化 JWT の取り扱い

機密パラメータ（OIDC の `claims` など）を保護する場合は JWE 暗号化を使用します。JWE の復号に失敗した場合は、プレーンテキストへのフォールバックを行わず、リクエストを即座に拒否してください。

## 採用事例

### FAPI 2.0 Security Profile

FAPI 2.0 Security Profile は金融 API のセキュリティ標準です。**Baseline Profile** では PAR（Pushed Authorization Requests）の使用が必須化されており、JAR はオプションです。Authorization Code Flow + PKCE + バックチャネル送信で十分なセキュリティを確保できるため、前チャネルでのリクエスト整合性保護の優先度が FAPI 1.0 から低下しました。

FAPI 1.0 では認可エンドポイントのフロントチャネルに ID Token が返るフローを使用していたため、リクエスト改ざん防止として JAR が必須でした。FAPI 2.0 はこの設計を刷新し、フロントチャネルには最小限の情報のみ渡す構成になっています。

**FAPI 2.0 Message Signing Profile** ではメッセージ整合性の保証が求められるユースケース向けに JAR を使用します。金融サービス・オープンバンキング API（英国 Open Banking、欧州 Berlin Group NextGenPSD2、オーストラリア CDR など）での実装が進んでいます。

### OpenID for Verifiable Presentations (OID4VP)

分散型アイデンティティ領域でも JAR の採用が進んでいます。OID4VP では Wallet に対する Presentation Request を JAR 形式で送信する構成が標準化されており、要求する Credential の詳細を署名付きリクエストオブジェクトに含めることで、リクエストの正当性を暗号的に保証します。

### ライブラリサポート

主要な OAuth ライブラリが RFC 9101 をサポートしています。

- **Authlib** (Python): RFC 9101 対応済み
- **Spring Security**: Spring Authorization Server で JAR サポート
- **Curity Identity Server**: FAPI 2.0 準拠として JAR + PAR を完全サポート
- **Duende IdentityServer**: JAR + PAR の組み合わせ実装を提供
