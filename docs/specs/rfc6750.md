# The OAuth 2.0 Authorization Framework: Bearer Token Usage (RFC 6750)

## 概要

RFC 6750 は、OAuth 2.0 で発行されたアクセストークンをリソースサーバーへのリクエストに付与する方法を規定しています。正式名称は "The OAuth 2.0 Authorization Framework: Bearer Token Usage"、2012 年 10 月に Proposed Standard として発行されました。

Bearer Token とは「持っている者は誰でも使える」トークンです。暗号鍵の所有証明を必要としないため、実装が極めてシンプルになります。その一方、トークンを盗まれた者は正規ユーザーと同じアクセス権を得てしまいます。この特性ゆえ、RFC 6750 は TLS（HTTPS）を絶対的な前提条件として位置づけ、「TLS があるからこそ Bearer Token が成立する」という設計になっています。

OAuth 2.0 エコシステムにおける役割分担を整理すると、RFC 6749 がトークン取得のフローを定義し、RFC 6750 がそのトークンの**使い方**を定義します。現在もほぼすべての OAuth 2.0 実装が RFC 6750 に準拠しており、2025 年 1 月発行の RFC 9700（OAuth 2.0 Security Best Current Practice）によって補足・更新されています。

## 背景と経緯

OAuth 1.0 では、API リクエストのたびに HMAC-SHA1 署名を生成する必要がありました。クライアントシークレット・リクエストトークン・タイムスタンプ・ノンスを組み合わせた複雑な署名処理は、実装者にとって大きな負担でした。

OAuth 2.0 の設計思想は「TLS を前提として署名処理を排除する」ことでした。HTTPS が普及しつつあった 2010 年代初頭、Bearer Token はその方向性を体現した仕様です。トークンを HTTP リクエストに付与するだけでよく、署名計算・ノンス管理・タイムスタンプ同期が不要になりました。

この設計は急速な普及を実現しました。Auth0、Okta、Microsoft Entra ID（旧 Azure AD）、Google、AWS など、主要なクラウドサービスはすべて RFC 6750 に準拠した Bearer Token を採用しています。

## 技術詳細

### Bearer Token の 3 つの送信方法

RFC 6750 は Bearer Token をリソースサーバーへ送信する 3 つの方法を定義しています（[RFC 6750 Section 2](https://www.rfc-editor.org/rfc/rfc6750#section-2)）。

#### 1. Authorization リクエストヘッダー（推奨）

```http
GET /resource HTTP/1.1
Host: server.example.com
Authorization: Bearer mF_9.B5f-4.1JqM
```

HTTP 標準の `Authorization` ヘッダーを使用します。リソースサーバーはこの方法を必ずサポートしなければなりません（MUST）。ブラウザの履歴やサーバーログに記録されにくく、セキュリティリスクが最も低い方法です。

#### 2. Form-Encoded ボディパラメーター（条件付き許容）

```http
POST /resource HTTP/1.1
Host: server.example.com
Content-Type: application/x-www-form-urlencoded

access_token=mF_9.B5f-4.1JqM&other_param=value
```

ブラウザが `Authorization` ヘッダーに直接アクセスできない環境を想定した代替手段です。使用条件は厳格で、POST/PUT などのボディを持つリクエストに限定され、`Content-Type: application/x-www-form-urlencoded` が必須です。GET リクエストへの使用は禁止されています。

#### 3. URI クエリパラメーター（非推奨）

```http
GET /resource?access_token=mF_9.B5f-4.1JqM HTTP/1.1
Host: server.example.com
```

RFC 6750 は「ほぼ不可能な場合を除き使用すべきでない（SHOULD NOT）」と明記しています（[Section 2.3](https://www.rfc-editor.org/rfc/rfc6750#section-2.3)）。理由は明確で、URL はブラウザの履歴、Web サーバーのアクセスログ、Referer ヘッダーを通じて漏洩する経路が多岐にわたるためです。開発時のデバッグ目的でも使用するべきではありません。

### エラーレスポンス

リソースサーバーはトークンが無効な場合に `WWW-Authenticate` ヘッダーでエラーを返します（[RFC 6750 Section 3](https://www.rfc-editor.org/rfc/rfc6750#section-3)）。

```http
HTTP/1.1 401 Unauthorized
WWW-Authenticate: Bearer realm="example",
                  error="invalid_token",
                  error_description="The access token expired"
```

エラーコードは 3 種類定義されています。

| エラーコード         | HTTP ステータス | 意味                                                         |
| -------------------- | --------------- | ------------------------------------------------------------ |
| `invalid_request`    | 400             | トークンが複数の方法で提供された、必須パラメーターの欠落など |
| `invalid_token`      | 401             | トークンの有効期限切れ・失効・不正                           |
| `insufficient_scope` | 403             | トークンのスコープが不足（権限なし）                         |

`WWW-Authenticate` ヘッダーの `scope` パラメーターには、アクセスに必要なスコープを列挙できます。クライアントはこれを読み取り、追加の権限をユーザーに求めることができます。

### トークンのライフサイクル管理

RFC 6750 はトークン自体の形式を規定しません。不透明な文字列（opaque string）でも、JWT（RFC 9068）でも構いません。リソースサーバーがトークンを検証できれば、形式は問いません。

トークン有効期間については、RFC 6750 Section 5.2 が「アクセストークン漏洩の影響を限定するため、できる限り短い有効期間にすべき」と規定しています（[Section 5.2](https://www.rfc-editor.org/rfc/rfc6750#section-5.2)）。具体的な時間は仕様で定められていませんが、業界慣行として 1 時間以下が一般的です。有効期限が切れたトークンはリフレッシュトークンで更新し、古いアクセストークンは再利用できません。

## 設計思想とトレードオフ

### シンプルさの代償

Bearer Token の最大の設計判断は「所有証明の省略」です。暗号鍵を持っていなくてもトークンさえあれば使えます。これにより、実装の容易さと幅広い採用を実現しましたが、根本的なリスクをはらんでいます。

- **トークン盗難 = 即時なりすまし**: 盗まれたトークンを持つ者は、正規ユーザーと同一のアクセス権を得ます
- **送信者の証明不可能**: トークンの正規所有者かどうかを確認する手段がありません
- **リプレイ攻撃の許容**: 同一トークンを別の宛先で再利用できます

これらのリスクはすべて TLS によって軽減されるという前提で設計されています。TLS がない環境での Bearer Token 使用は設計上考慮されていません。

### なぜ今でも標準なのか

2012 年の仕様でありながら RFC 6750 が現役である理由は、シンプルさと相互運用性の組み合わせが実用的だからです。DPoP（RFC 9449）や mTLS（RFC 8705）のような送信者制約付きトークンはより安全ですが、実装の複雑さが増します。RFC 9700 は高セキュリティ要件では DPoP や mTLS を推奨しつつも、Bearer Token 自体を廃止してはいません。

## DPoP との比較

[RFC 9449](rfc9449.md)（Demonstrating Proof of Possession）は Bearer Token の根本的な弱点——送信者証明の欠如——を解決するために設計されました。

| 項目           | Bearer Token (RFC 6750)      | DPoP (RFC 9449)                         |
| -------------- | ---------------------------- | --------------------------------------- |
| 所有証明       | なし（所持者誰でも使用可）   | あり（秘密鍵保有者のみ使用可）          |
| 盗難時のリスク | 即座に悪用可能               | 秘密鍵がなければ使用不可                |
| 実装複雑性     | 低い                         | 高い（非対称鍵管理が必要）              |
| パフォーマンス | 高い                         | やや低い（リクエストごとに署名計算）    |
| 適した環境     | サーバー間通信、低リスク環境 | SPA、ブラウザアプリ、高セキュリティ要件 |

FAPI 2.0 のような高セキュリティプロファイルでは、DPoP または mTLS による送信者制約付きトークンが必須または強く推奨されています（[FAPI 2.0 Security Profile](fapi2.md)参照）。

## 実装上の注意点

### TLS の完全な検証

「HTTPS を使っているから安全」は不十分です。以下をすべて確認する必要があります。

- TLS 1.2 以上（TLS 1.0/1.1 は RFC 8996 で廃止）
- サーバー証明書チェーンの検証
- ホスト名の検証
- 証明書失効確認（CRL または OCSP）

TLS 証明書の検証を省略した実装は、中間者攻撃によりトークンが丸ごと盗まれます。

### リダイレクト時のトークン漏洩

HTTP リダイレクトが発生すると、`Authorization` ヘッダーが意図しないホストに送られるリスクがあります。特に異なるオリジンへのリダイレクトや、クロスプロトコルリダイレクト（HTTP から別プロトコルへの転送）では Bearer Token が漏洩する危険性があります。リソースサーバーへのリクエストは不必要なリダイレクトを発生させない設計が重要です。

### スコープとオーディエンスの制限

トークンは必要最小限のスコープで発行すべきです（[RFC 6750 Section 5.2](https://www.rfc-editor.org/rfc/rfc6750#section-5.2)）。JWT アクセストークンを使用する場合は RFC 9068 に従い `aud`（audience）クレームで対象リソースサーバーを制限することが推奨されます（[RFC 9068 Section 4](https://www.rfc-editor.org/rfc/rfc9068#section-4)）。スコープもオーディエンスも制限しないトークンは、漏洩した際に全リソースへのアクセスを許してしまいます。

### URI Query Parameter の絶対的な回避

たとえデバッグ目的であっても `?access_token=...` 形式は使用しないでください。開発環境のログが本番のトークンで汚染され、そのログが意図せず共有されるケースは珍しくありません。
