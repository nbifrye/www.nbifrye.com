# /review — 既存コンテンツのレビュー・改善

このスキルは人間の介入なしに完走するよう設計されています。
不明点があっても人間に質問せず、最善判断で進めてください。

まず `CLAUDE.md` と `.claude/skills/work/style-guide.md` を読んでプロジェクト全体の方針を把握してください。

---

## Step 1 — 対象ファイルの選定

### 1-1. 既存ファイルを列挙する

```
Glob("docs/articles/*.md") → index.md 以外を抽出
Glob("docs/specs/*.md")    → index.md 以外を抽出
```

### 1-2. 未レビューのファイルを特定する

各ファイルのフロントマターを Read ツールで確認し、以下の条件で絞り込む:

- `tags` フィールドに `レビュー済み` が含まれているファイルは除外する
- 該当しないファイル（未レビュー）を候補とする

**全件レビュー済みの場合**: 「すべての記事はレビュー済みです」と報告して終了する。

### 1-3. レビュー対象を 1 件選ぶ

未レビューのファイルが複数ある場合、以下の優先順位で 1 件を選ぶ:

1. Articles より Specs を優先（より事実確認が重要なため）
2. 同種類の中では、ファイル名のアルファベット順で最初のもの

選定したファイルと選定理由を 1〜2 文で記録してください。

---

## Step 2 — 正確性レビュー

Explore サブエージェントを起動し、一次情報と照合した技術的正確性のレビューを行ってください。

```
Agent(subagent_type="Explore", prompt="""
以下のファイルを技術的正確性の観点でレビューしてください。

## 対象ファイル

<ファイルパス>

## 手順

1. 対象ファイルを Read ツールで全文読む
2. 参考資料セクションの URL を WebFetch で取得し、以下を確認:
   - 仕様バージョンは最新か（URL が示す文書のバージョン番号と記事の記載を照合）
   - 仕様ステータスは正確か（Draft / Final / Superseded）
3. 本文中の主要な技術的主張（3〜5 件）を選び、一次ソースとの整合性を確認
4. 参考資料 URL が存在しない主張がある場合は指摘する

## チェック項目

- [ ] 仕様バージョンが最新（古い場合は "as of <version>" として明記すべき）
- [ ] 全ての技術的主張に根拠（URL・RFC番号）が示されている
- [ ] 誤解を招く表現・事実誤認がない
- [ ] 関連仕様との整合性（矛盾がない）
- [ ] セキュリティ上の注意点が適切に記載されている

## 返答形式

問題なし: 「レビュー完了：重大な問題なし」
問題あり: 問題箇所・修正案を箇条書きで（ファイルの行番号または引用文付き）
""")
```

## Step 3 — 可読性レビュー

対象ファイルを Read ツールで全文読み、以下の観点でチェックしてください。

- **文末統一**: 「です・ます」調（敬体）に統一されているか。混在していれば修正候補を特定する
- **冗長表現**: 同じ内容の繰り返し・余分な接続詞（「また、」「なお、」の連続など）
- **句読点**: 長い文（60字超）を読みやすく分割できるか
- **受動態**: 「〜されています」「〜と言われています」の多用 → 能動態に書き換えられるか
- **用語表記の揺れ**: `.claude/skills/work/style-guide.md` の表記ルールに照らして揺れがないか

---

## Step 4 — ファイル修正

### 4-1. 本文の修正

Step 2・Step 3 で発見した問題を Edit ツールで修正してください。

修正の優先順位:

1. 事実誤認・バージョン誤記（必ず修正）
2. 誤解を招く表現（必ず修正）
3. 文末の不統一（修正）
4. 冗長表現・受動態（軽微であれば修正、大幅な書き換えが必要な場合はスキップ）

**注意**: タイトル（`# 見出し`）や AI 免責 Note の文言は変更しない。

### 4-2. フロントマターへのタグ追加

対象ファイルのフロントマターに `tags` フィールドを追加してください。

**Articles** (`docs/articles/YYYY-MM-DD-slug.md`) の場合、`date` フィールドの直後に追加:

```yaml
---
title: ...
description: ...
date: YYYY-MM-DD
tags:
  - レビュー済み
---
```

**Specs** (`docs/specs/id.md`) の場合、`description` フィールドの直後に追加:

```yaml
---
title: ...
description: ...
tags:
  - レビュー済み
---
```

PostToolUse フックが自動的に oxfmt でフォーマットします。

## Step 5 — ビルド検証

コミット前に VitePress ビルドが通ることを確認してください。

```bash
npm run docs:build 2>&1
```

ビルドエラーが出た場合は以下を確認して修正し、再ビルドしてください:

- frontmatter の構文エラー（引用符の欠落・インデントの誤り）
- Markdown の見出し構造（h1 が複数ある等）
- 内部リンク切れ

---

## Step 6 — Git commit & push

今回変更したファイルのみをステージしてコミット・プッシュしてください。

```bash
# 変更したファイルのみ追加
git add <対象ファイルパス>

# コミットメッセージ
git commit -m "chore: <記事タイトルの要約>をレビュー・改善"

# main ブランチへの直接 push を試みる
git push origin HEAD:main
```

**push 結果に応じた対処:**

- **成功**: そのまま終了
- **失敗**（ブランチ保護・競合など）: 現在のブランチへ push する

```bash
git push -u origin HEAD
```

---

## エラー対処

- **WebFetch 失敗**: 別の URL を試す（HTML 版 → テキスト版 RFC、別ミラーなど）
- **ビルドエラー**: frontmatter の構文エラー・リンク切れを修正してから再実行
- **push 失敗（競合）**: `git pull --rebase origin main` してから再試行
- **push 失敗（ブランチ保護）**: 現在のブランチへ push して終了
