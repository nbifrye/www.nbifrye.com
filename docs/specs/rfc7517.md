# JWK — JSON Web Key (RFC 7517)

## 概要

**JSON Web Key (JWK)** は、暗号鍵を JSON オブジェクトとして表現するための標準形式を定義した IETF 仕様（[RFC 7517](https://www.rfc-editor.org/rfc/rfc7517.html)）である。2015 年 5 月に Proposed Standard として公開され、JWT (RFC 7519)・JWS (RFC 7515)・JWE (RFC 7516)・JWA (RFC 7518) とともに **JOSE (JavaScript Object Signing and Encryption)** ファミリーを構成する。

JWK が解決する問題は「鍵の相互運用可能な表現形式の欠如」である。それ以前、RSA 公開鍵は PEM や DER、X.509 証明書など複数の形式が混在しており、Web API での鍵交換に統一的な方法がなかった。JWK は鍵タイプ・用途・アルゴリズム・識別子をすべて単一の JSON オブジェクトに収め、HTTP で容易に配信できる形式を提供した。

OpenID Connect Discovery では IdP が `jwks_uri` を公開し、クライアントが ID Token の署名検証鍵を動的に取得する仕組みがある。この仕組みの根幹を支えるのが JWK と JWK Set である。

## 背景と経緯

JOSE ファミリーの設計は 2012 年頃から IETF の JOSE WG で進められた。当時の Web セキュリティは XML Signature・XML Encryption（SAML が依拠する仕様群）を前提としており、JSON ベースのトークン（OAuth 2.0 のアクセストークン等）に対応する軽量な暗号フレームワークが求められていた。

RFC 7517 は RFC 7515〜7519 とほぼ同時に策定・発行された。著者の Michael B. Jones（当時 Microsoft）は JOSE WG の中心人物として一連の仕様設計を主導した。

JOSE 仕様群のうち JWK は「鍵の表現」という最も基礎的な役割を担い、他の仕様はすべて JWK を前提として動作する。

## 設計思想

### JSON ファーストの鍵表現

XML ベースの鍵表現（KeyInfo、KeyValue など）に対し、JWK は JSON のみで完結する。HTTP API での利用を前提とした設計であり、Base64url エンコードを用いてバイナリデータを JSON 文字列として表現する。

### 拡張可能なキータイプ登録

`kty` パラメータは IANA レジストリ（[JSON Web Key Types](https://www.iana.org/assignments/jose/jose.xhtml#web-key-types)）で管理される。RFC 7517 発行時点では RSA・EC・oct の 3 種が定義されており、後から RFC 8037 で OKP（Ed25519/X25519）が追加された。新しいアルゴリズムが登場しても既存の仕様を改訂せず IANA 登録で対応できる設計になっている。

### 用途と操作の分離

`use`（公開鍵用途）と `key_ops`（許可操作のリスト）は設計上異なるレイヤーを担う。`use` は「署名用か暗号化用か」という高レベルな区分であり、`key_ops` は「sign・verify・encrypt・decrypt・wrapKey・unwrapKey」などのより細粒度な操作を指定する。仕様は `use` と `key_ops` を同時に使う場合は整合性を保つよう求めている。

### JWK Set による複数鍵管理

単一の JWK だけでなく、複数の JWK を `keys` 配列にまとめた **JWK Set** を定義している点が重要な設計判断である。これによりキーローテーション中に旧鍵・新鍵を同時公開できる。移行期間中の互換性維持が要件として織り込まれた設計といえる。

## 技術詳細

### JWK の共通パラメータ

すべての JWK に適用可能なパラメータは以下のとおり。

| パラメータ | 型     | 必須 | 説明                                               |
| ---------- | ------ | ---- | -------------------------------------------------- |
| `kty`      | 文字列 | ○    | 鍵タイプ（RSA / EC / oct / OKP）                   |
| `use`      | 文字列 | —    | 用途（`"sig"` = 署名、`"enc"` = 暗号化）           |
| `key_ops`  | 配列   | —    | 許可操作（sign / verify / encrypt / decrypt など） |
| `alg`      | 文字列 | —    | 使用アルゴリズム識別子（RFC 7518 のレジストリ値）  |
| `kid`      | 文字列 | —    | 鍵識別子                                           |
| `x5u`      | URI    | —    | X.509 証明書チェーンへの参照 URI                   |
| `x5c`      | 配列   | —    | Base64 エンコードされた証明書チェーン              |
| `x5t`      | 文字列 | —    | X.509 証明書 SHA-1 サムプリント                    |
| `x5t#S256` | 文字列 | —    | X.509 証明書 SHA-256 サムプリント                  |

`kty` のみが必須であり、他はすべてオプショナルである。ただし実運用では `kid` の付与が事実上必須となる。

### 鍵タイプ別パラメータ

#### RSA 鍵

```json
{
  "kty": "RSA",
  "use": "sig",
  "alg": "RS256",
  "kid": "2024-rsa-01",
  "n": "0vx7agoebGcQSuuPiLJXZptN9nndrQmbXEps2aiAFb...",
  "e": "AQAB"
}
```

公開鍵パラメータは `n`（モジュラス）と `e`（公開指数）。秘密鍵はさらに `d`（秘密指数）、`p`・`q`（素因数）、`dp`・`dq`（CRT 指数）、`qi`（CRT 係数）を含む。秘密鍵パラメータが含まれる JWK は絶対に公開エンドポイントに露出させてはならない。

#### EC 鍵（楕円曲線）

```json
{
  "kty": "EC",
  "crv": "P-256",
  "use": "sig",
  "kid": "2024-ec-01",
  "x": "f83OJ3D2xF1Bg8vub9tLe1gHMzV76e8Tus9uPHvRVEU",
  "y": "x_FEzRu9m36HLN_tue659LNpXW6pCyStikYjKIWI5a0"
}
```

`crv` で楕円曲線を指定（P-256 / P-384 / P-521）。秘密鍵の場合は `d` パラメータが追加される。

#### OKP 鍵（Ed25519/X25519）— RFC 8037

```json
{
  "kty": "OKP",
  "crv": "Ed25519",
  "use": "sig",
  "x": "11qYAYKxCrfVS_7TyWQHOg7hcvPapiMlrwIaaPcHURo"
}
```

RFC 8037 で追加されたキータイプ。EdDSA 署名（Ed25519/Ed448）と ECDH 鍵交換（X25519/X448）に対応する。

#### 対称鍵（oct）

```json
{
  "kty": "oct",
  "alg": "HS256",
  "k": "GawgguFyGrWKav7AX4VKUg"
}
```

HMAC や AES などの対称鍵を Base64url エンコードで表現する。

### JWK Set (JWKS)

複数の JWK を `keys` 配列にまとめた JSON オブジェクト。OpenID Connect Discovery における `jwks_uri` エンドポイントはこの形式でレスポンスを返す。

```json
{
  "keys": [
    {
      "kty": "RSA",
      "use": "sig",
      "kid": "2024-01",
      "n": "...",
      "e": "AQAB"
    },
    {
      "kty": "RSA",
      "use": "sig",
      "kid": "2024-02",
      "n": "...",
      "e": "AQAB"
    }
  ]
}
```

キーローテーション中は旧鍵と新鍵を同時にセット内に保持する。クライアントは JWT ヘッダの `kid` と照合して検証鍵を選択する。

### JWK Thumbprint（RFC 7638）

[RFC 7638](https://www.rfc-editor.org/rfc/rfc7638.html) は JWK の正規化ハッシュ（通常 SHA-256）を定義する。計算手順は：

1. 必須パラメータのみ（`kty`・鍵タイプ固有の必須フィールド）を辞書順にソートして JSON シリアライズ
2. SHA-256 でハッシュ化
3. Base64url エンコード

オプショナルなメタデータ（`use`・`alg`・`kid` など）は計算に含まれない。これにより同じ鍵の公開形式と秘密形式が同一の Thumbprint を持つ特性が実現する。`kid` の値として JWK Thumbprint を使うことが推奨されており、これにより `kid` の衝突リスクを根本的に回避できる。

[RFC 9278](https://www.rfc-editor.org/rfc/rfc9278.html) では Thumbprint を URI スキームとして表現する形式（`urn:ietf:params:oauth:jwk-thumbprint:sha-256:...`）が追加されている。

### OpenID Connect Discovery との連携

OpenID Connect Discovery 1.0 では、IdP が `/.well-known/openid-configuration` で `jwks_uri` を公開する。クライアントはこの URI から JWKS を取得して ID Token の署名を検証する。

```
GET /.well-known/openid-configuration
→ { "jwks_uri": "https://example.com/.well-known/jwks.json", ... }

GET /.well-known/jwks.json
→ { "keys": [...] }
```

クライアントは検証に失敗した場合、`kid` が一致するキーが JWKS に存在しない場合に限り JWKS を再取得してよい。無制限な再取得はサービス拒否につながるため、キャッシュ戦略が重要になる。

## 実装上の注意点

### アルゴリズム混乱攻撃

RS256 で署名された JWT を HS256 として検証し、RSA 公開鍵を HMAC シークレットとして悪用する攻撃が知られている（[PortSwigger Web Security Academy](https://portswigger.net/web-security/jwt/algorithm-confusion)）。対策として、検証ライブラリに期待するアルゴリズムを明示的に指定し、JWK の `alg` パラメータのみを信頼しない実装が求められる。

### `kid` 衝突問題

複数の IdP やサービスが同一の `kid` 値を使う場合、誤ったキーで署名検証が成功する可能性がある。対策として JWK Thumbprint（RFC 7638）を `kid` 値として採用するか、`x5t#S256` による証明書ベースの識別を用いる。

### `x5u` パラメータの取り扱い

`x5u` パラメータは証明書チェーンへの URI を指す。攻撃者が制御するサーバへの URI が埋め込まれる可能性があるため、事前に信頼された URI のみを受け入れるホワイトリスト検証が必要である。

### JWKS のキャッシュと再取得

JWKS を過剰に頻繁に再取得する実装は IdP への DoS になりうる。逆にキャッシュが長すぎるとキーローテーション後も旧鍵で検証しようとして失敗し続ける。検証失敗かつ `kid` 不一致のときのみ再取得する設計が推奨される。再取得回数には上限を設けること。

### 秘密鍵パラメータの漏洩

`d`（RSA 秘密指数 / EC 秘密値）を含む JWK を誤って公開エンドポイントに露出させると即座の鍵漏洩となる。JWK Set の公開エンドポイントには公開鍵パラメータのみが含まれることを実装レベルで保証する必要がある。RFC 7517 セクション 9 は「秘密鍵と対称鍵は意図しない関係者への開示から保護されなければならない」と明記している。

## 採用事例

JWK / JWKS は OpenID Connect エコシステムで事実上必須の基盤技術となっている。

- **Google**: `https://www.googleapis.com/oauth2/v3/certs` で JWKS を公開し、Google Sign-In の ID Token 検証に利用
- **Microsoft**: Azure AD / Entra ID が `jwks_uri` 経由で署名検証鍵を公開
- **Auth0**: 自動キーローテーションを実装し、アクティブキーと次期キーを同時に JWKS で提供
- **Okta**: `/keys` エンドポイントで JWKS を公開し、2048 ビット RSA キーを標準採用
- **DPoP (RFC 9449)**: クライアント証明書の代わりに JWK を使い、アクセストークンを特定鍵にバインドする仕組みで JWK Thumbprint を活用
