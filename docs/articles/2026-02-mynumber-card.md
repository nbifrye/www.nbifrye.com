# マイナンバーカード技術解説——JPKI・スマホ対応・次期カードまで

> 日本の全国民向けデジタルIDインフラとして進化するマイナンバーカードを、技術仕様・最新動向・国際的文脈から体系的に整理する。

## はじめに

マイナンバーカード（個人番号カード）は、日本の公的個人認証基盤（JPKI: Japanese Public Key Infrastructure）の物理的な起点だ。2016年1月の交付開始から10年を経て、**2025年12月3日に交付枚数が1億枚を突破**（普及率80.3%）した。保険証廃止・免許証一体化・スマホへの電子証明書搭載と、この1〜2年で活用領域が一気に拡大した。

この記事では、マイナンバーカードの技術的な構造から、2025〜2026年の最新動向、2028年に控える次世代カードの設計、そして EU EUDI Wallet などグローバルな動向との比較まで、デジタルアイデンティティの観点から整理する。

---

## ICチップの技術構造

### 搭載されている4つのアプリケーション

マイナンバーカードのICチップには、物理的に区切られた4つのアプリケーション領域が存在する。

| アプリケーション | 内容 | 閲覧条件 |
|----------------|------|---------|
| **券面事項入力補助AP** | 氏名・住所・生年月日・性別・マイナンバー | 暗証番号（数字4桁） |
| **住民基本台帳AP** | 住民票コード | 暗証番号 |
| **公的個人認証AP（利用者証明用）** | 利用者証明用電子証明書 | 暗証番号（数字4桁） |
| **公的個人認証AP（署名用）** | 署名用電子証明書 | 暗証番号（英数字6〜16桁） |

このうちデジタルアイデンティティの観点で重要なのは「公的個人認証AP」の2種類の電子証明書だ。

### 2種類の電子証明書

**利用者証明用電子証明書**はオンライン認証に使う。「あなたが本人である」という事実を証明するもので、氏名・住所・生年月日・性別・マイナンバーは含まれない。マイナポータルへのログインや健康保険証としての利用で使われる。

**署名用電子証明書**は電子文書への署名に使う。こちらは氏名・住所・生年月日・性別（基本4情報）が含まれる。確定申告（e-Tax）や行政手続きでの電子署名で利用される。住所変更等の基本4情報に変化があると失効する仕組みになっている。

### 現行の暗号仕様（RSA-2048）

現行カードの電子証明書は **RSA-2048 / SHA-256** ベースだ。認証局（CA）は **J-LIS（地方公共団体情報システム機構）** が運営しており、国家的なトラストアンカーとして機能する。

電子証明書の有効期間は発行時に18歳以上の場合は **5年**。カード本体の有効期間は10年（18歳未満は5年）のため、カードの有効期間内に電子証明書を1〜2回更新する必要がある。これが後述する「証明書一斉期限切れ問題」の根源となっている。

## 最新動向：2025年に集中した活用拡大

### マイナ保険証への完全移行

2024年12月2日、従来の健康保険証の新規発行が停止され、マイナンバーカードを使った「マイナ保険証」を基本とする体制に移行した。2025年12月2日をもって、既存の保険証の有効期限が満了。以降は以下の3択になった。

1. **マイナ保険証**：マイナンバーカードを医療機関のリーダーにかざして認証
2. **スマホマイナ保険証**（2025年9月19日〜）：スマホでICチップに代わる認証
3. **資格確認書**：マイナンバーカードを持たない人向けの紙の代替証

2025年10月時点でマイナ保険証の利用率は国全体で **37.14%** に達した（2024年10月時点の利用件数は約3,400万件、1年前比5倍以上）。ただし紙の保険証の同月利用件数は約1億8,000万件であり、まだ移行途上にある。

2026年3月末まで、期限切れ保険証でも受診できる暫定措置が設けられているが、これはあくまで猶予期間だ。

### マイナ免許証：運転免許証との一体化

2025年3月24日、マイナンバーカードに運転免許証の情報を格納できる「マイナ免許証」の全国運用が始まった（道路交通法改正・令和4年）。2025年10月末時点の保有者数は **178万人**。

- 「マイナ免許証のみ」「2枚持ち」「従来の免許証のみ」の3択で任意選択
- 優良運転者がマイナ免許証を選ぶとオンライン講習で手数料が2,300円（従来3,350円）
- 2025年9月1日〜：カード更新時に免許情報が新カードへ自動引き継ぎ
- 従来の免許証も2027年まで並行利用可能

### スマホ対応：AndroidとiPhoneで異なる2つのアーキテクチャ

マイナンバーカードのスマホ対応は、AndroidとiPhoneで根本的にアーキテクチャが異なる点が技術的に重要だ。

**Android（スマホJPKI）——2023年5月〜**

Android向けは **GlobalPlatform（GP）準拠のSecure Element（GP-SE）** にJPKIアプレットをインストールする実装だ。

- **格納場所**：端末内の耐タンパ性ハードウェア（GP-SE）。おサイフケータイ対応端末限定
- **TSM**：フェリカネットワークスがTSM（Trusted Service Manager）を担当
- **セキュアチャネル**：サーバ〜GP-SE間の通信はSCP03（GlobalPlatformセキュアチャネルプロトコル）
- **証明書形式**：物理カードと同じJPKI-AP（X.509、RSA-2048）を搭載
- **生体認証連携**：PINをAndroid OSのBiometricPromptで代替可能
- **2026年秋頃**：「Androidのマイナンバーカード」として刷新予定（年齢確認・属性証明機能を追加）

**iPhone（iPhoneのマイナンバーカード）——2025年6月24日〜**

iPhoneの実装はAndroidと全く異なる。**Apple Walletに ISO/IEC 18013-5（mdoc）形式**で格納される。Appleが国家IDをWalletに搭載する世界初の事例。

- **格納場所**：iPhoneの Secure Enclave（ハードウェアセキュリティ領域）
- **データ形式**：ISO/IEC 18013-5準拠のmdoc（CBOR形式）
- **格納情報**：氏名・住所・生年月日・性別・マイナンバー・顔写真・J-LIS発行の電子署名
- **対面提示**：NFC/QRコード（ISO/IEC 18013-5 DeviceAuth準拠）
- **オンライン提示**：Apple ID Verifier APIを使用
- **生体認証**：Face IDまたはTouch IDで認証
- **対応機種**：iPhone XS/XR以降、iOS 18.5以降

この二分化は見落としやすいが本質的な違いだ。Android JPKIは「物理カードのデジタル複製（X.509 PKI）」なのに対し、iPhoneは「国際標準のVerifiable Credentials（mdoc）」として実装されている。iPhoneのマイナンバーカードは EUDI Wallet や海外mDLと同じ技術スタックに乗っており、グローバルな相互運用性の起点になりうる。

2025年時点でスマホ用電子証明書を取得して活用している人は人口の **約2.7%** に留まっており、存在を知られていない段階だ。

### デジタル認証アプリ——OpenID Connectで民間に開く

**2024年6月24日**にデジタル庁がリリースした「デジタル認証アプリ」は、マイナンバーカードを使ったOpenID Connect / OAuth 2.0準拠のIdP（Identity Provider）だ。

従来のJPKI連携は「認定事業者のみ」「専用SDK必須」だったが、デジタル認証アプリにより **OpenID Connect標準に準拠したRPなら誰でも連携可能** になった。

| 機能 | 仕様 |
|------|------|
| 認証API | JPKIの署名用電子証明書でJ-LISサーバが有効性確認 → OIDCトークン発行 |
| 署名API | RP側でSHA-256ハッシュ生成 → DigestInfo付加でリクエスト |
| 通信 | HTTPS（TLS 1.2/1.3） |
| フィッシング耐性 | 正規URLでのみ動作 |

2025年10月時点で **857社**（3年前の約5倍）の民間事業者がマイナンバーカードを活用している。2025年9月以降はマイナンバーのスコープも追加された。

### 電子証明書一斉期限切れ問題

2021年前後のマイナポイント特需期に発行されたカードの電子証明書が、2026年以降に一斉に有効期限を迎える。総数は **2026年度だけで1,500万件超、2027年度以降は年間約3,000万件** に急増する見込み。

有効期限の更新には市区町村窓口への来庁が基本的に必要（一部オンライン更新も整備中）であり、窓口が飽和するリスクが指摘されている。さらに、カードと電子証明書の有効期限が異なることを把握している人は調査上 **約48.6%** に留まる。

## 次世代カードへの移行：2028年度

デジタル庁は2024年3月に「次期個人番号カードタスクフォース最終とりまとめ」を公表し、2028年度を目標とした次世代カードの設計を明示した。

### 暗号仕様の刷新：RSA → ECDSA

最大の技術的変化は暗号方式の更新だ。

| 項目 | 現行カード | 次期カード |
|------|-----------|-----------|
| 公開鍵暗号 | RSA-2048 | ECDSA P-384（必須）/ ECDSA P-521（オプション） |
| ハッシュ | SHA-256 | SHA-384（必須）/ SHA-512（オプション） |
| セキュリティ強度 | 112ビット | 192〜256ビット |
| PQC対応 | なし | 将来対応を検討 |

ECDSA への移行は、鍵長を縮小しながらセキュリティ強度を高める効果がある。特に P-521 オプションで256ビットセキュリティに対応しておくことで、量子コンピュータ時代に向けた PQC（耐量子計算機暗号）移行の準備が整う設計だ。

### 利便性改善

| 項目 | 現行カード | 次期カード |
|------|-----------|-----------|
| 電子証明書有効期間 | 5年 | **10年**（成人） |
| 暗証番号の種類 | 最大4種類 | **2種類**に統合 |
| APDU仕様書 | 非公開 | **公開**（開発者向け） |
| 更新申請可能時期 | 3ヵ月前 | **1年前**から |
| 券面の性別欄 | あり | **廃止**（ICチップ内にのみ格納） |

電子証明書の有効期間がカード本体（10年）に揃うことで、現在の「カードの有効期間中に何度も証明書更新が必要」という手間が解消される。

APDU仕様書の公開は特に重要だ。現行カードではインターフェイス仕様が非公開（認定事業者のみ開示）のため、マイナンバーカードを使う端末・アプリの開発コストが高かった。次期カードで仕様を公開することで、民間の開発参入障壁が下がる。

## JPKIとグローバルスタンダードの関係

### EU EUDI Wallet との比較

マイナンバーカード（JPKI）と EU EUDI Wallet を比較すると、設計思想の共通点と差異が見えてくる。

| 観点 | JPKI（日本・物理カード + Android） | iPhoneのマイナンバーカード | EUDI Wallet（EU） |
|------|-----------------------------------|--------------------------|-----------------|
| トラストアンカー | J-LIS（国家CA） | J-LIS（国家CA） | EU加盟国の政府CA |
| モバイル実装 | GP-SE（おサイフケータイ必須） | Apple Wallet / Secure Enclave | モバイルウォレット |
| クレデンシャル形式 | X.509 PKI 証明書（RSA-2048） | **ISO 18013-5 mdoc** | SD-JWT VC / mdoc |
| オンライン連携 | デジタル認証アプリ（OpenID Connect） | Apple ID Verifier API | OpenID4VP / HAIP 1.0 |
| 選択的開示 | なし（バイナリ認証） | mdoc属性単位（部分対応） | SD-JWT / ZKP対応 |
| クエリ言語 | N/A | N/A | DCQL（HAIP 1.0） |

EUDI Wallet は OpenID4VP・OpenID4VCI・HAIP 1.0 という国際標準スタックを採用し、**選択的開示**（必要な属性だけを提示する能力）を基本設計に組み込んでいる。物理カード・Android JPKIは現状「認証する・しない」のバイナリな判定が中心だが、iPhoneのマイナンバーカードが ISO 18013-5 mdoc を採用したことで、EU・米国・日本のmDL実装と同じ技術スタックに乗ることになった。

日本とEUは **2025年度中に教育クレデンシャルに係る国際相互運用実証** を開始する計画（2025年6月閣議決定「デジタル社会の実現に向けた重点計画」）。また **2027年度内に行政分野での初VC活用ユースケース** の創出を目標としている。

### Digital Credentials API との連携可能性

W3C Digital Credentials API（Chrome 141・Safari 26 で出荷済み）は、ブラウザを通じてウォレットからクレデンシャルを提示するための API だ。同 API は OpenID4VP と ISO 18013-7 Annex C（mdoc）を normative reference として採用している。

現行のスマホJPKI（Android）は OpenID4VP との互換性を持たないが、iPhoneの mdoc 実装は ISO 18013-7 Annex C 経由でDC APIに対応できる理論的な道がある。Appleが公式のID Verifier APIをDC API準拠に拡張すれば、ブラウザ上でのマイナンバーカード提示が現実になる。

デジタル認証アプリはOpenID Connectを採用しているが、OpenID4VCというVerifiable Credentialsレイヤーへの対応はまだだ。デジタル庁がHAIP 1.0 の参照国（38法域）に含まれている以上、次期カードの設計にOpenID4VP対応を盛り込む選択肢は十分にある。

## 所感

マイナンバーカードはこの2年で「持っているだけ」から「日常的に使うもの」へと急速に転換した。保険証廃止・免許証一体化・スマホ搭載と、使われ方が急拡大している。しかし、ICチップの技術仕様は10年前のままで、国際的なデジタルアイデンティティ標準とのギャップが広がっている。

2028年の次期カードは、ECDSA への暗号更新・APDU仕様書の公開・暗証番号統合という実務的な改善が中心だが、より根本的な問いに答えていない。日本の JPKI をグローバルな OpenID4VP / HAIP 1.0 のエコシステムに接続するのか、日本独自の閉じたスタックで維持するのか、という選択だ。

EU EUDI Wallet は2026年12月に義務的展開が始まる。その頃、日本でも次期カードの設計が最終化される。グローバルスタンダードとの整合性を今の段階で設計に盛り込まないと、国際的な相互運用は日本だけの宿題として残り続ける。
