# 第2章 デジタルアイデンティティの基本用語

IAMを設計・運用する際、最初に詰まるのは技術そのものより**用語のズレ**です。
同じ「ID」という言葉でも、業務部門は「社員番号」、開発は「ユーザーID」、セキュリティは「認証情報」を指してしまい、要件定義が破綻しがちです。

私の実務感では、IAMプロジェクト初期の失敗の多くは「製品選定」ではなく、**Identity / Identifier / Credential / Claim を分けて話せないこと**から始まります。

## 1. 概念

まずは4つを明確に分けます。

1. **Identity（アイデンティティ）**: 主体（人・法人・端末・ワークロード）を表す情報のまとまり
2. **Identifier（識別子）**: 主体を一意に識別するキー（例: `sub`, 社員番号, 顧客ID）
3. **Credential（認証情報）**: 主体性を証明する材料（パスワード、秘密鍵、FIDO鍵、証明書）
4. **Claim（主張属性）**: 主体に関する属性情報（部署、年齢確認済み、権限ロールなど）

重要なのは、**Identifierは変更されにくい内部キーとして扱い、表示名やメールアドレスをID本体にしない**ことです。

## 2. なぜ必要か

用語を分けないまま設計すると、次の不具合が高確率で起きます。

- メールアドレス変更でアカウント同一性が壊れる
- 「認証済み」と「業務上許可される」を混同し、認可バグを生む
- 属性更新の遅延で、異動後も旧権限が残る
- 監査で「誰が何をしたか」を一意に説明できない

特にクラウド連携では、IdP・SaaS・社内アプリでIdentifierが不統一だと、退職者の一括剥奪や証跡突合に大きなコストがかかります。

## 3. 代表的な構成・流れ

用語を実装に落とす最小パターンは次の通りです。

1. IdPで認証し、**不変な主体ID（`sub`）**を発行する
2. アプリ側は`sub`を内部主キーとして紐付ける
3. 表示用ID（メール、社員番号）は別属性として保持する
4. 認可はロール/属性（Claim）で判定し、Credentialの強度とは分離する
5. 監査ログは`subject_id`を中心に保存し、属性変化履歴を残す

一次情報として以下を読むと、用語の境界が明確になります。

- OpenID Connect Core 1.0（`sub` Claimの定義）  
  <https://openid.net/specs/openid-connect-core-1_0.html>
- RFC 7519: JSON Web Token (JWT)（クレームモデル）  
  <https://datatracker.ietf.org/doc/html/rfc7519>
- NIST SP 800-63-4（Digital Identity Guidelines）  
  <https://pages.nist.gov/800-63-4/>

## 4. よくある失敗

### 失敗1: メールアドレスを不変IDとして使う

メールは運用上変更されます。人事改姓、組織変更、ドメイン統合で値が変わるため、内部キーに置くと参照整合性が壊れます。

### 失敗2: 認証と認可を同じ判定にしてしまう

「ログインできた」ことと「承認操作できる」ことは別です。Credential強度（MFA有無）と業務権限を分離しないと、権限昇格が起きやすくなります。

### 失敗3: Claimの信頼境界を曖昧にする

外部IdPの属性をそのまま業務権限へ反映すると、属性同期遅延や誤設定で事故が起きます。どの属性を誰がマスター管理するか明文化が必要です。

## 5. 日本での注意点

日本企業では、制度対応と運用都合でIdentifier設計が複雑化します。

- 人事番号・従業員番号・グループ会社IDが並立しやすい
- 委託先/派遣社員の識別子管理が本社員と異なる
- APPI対応で属性の利用目的・保持期間を説明できる設計が求められる
- 金融/通信/医療などでは本人確認結果（eKYC結果）の再利用条件が厳密

実務上は、**全システム共通のグローバル主体ID**と、システムローカルIDの対応表（identity mapping）を最初に作ることが効果的です。これがあるだけで監査・事故対応の初動速度が大きく変わります。

## 6. 最低限覚える語彙

- **Subject**: 認証・認可の対象主体
- **`sub`**: OIDCにおける主体識別子（RP内で一意）
- **Credential Binding**: 資格情報を主体へ結びつけること
- **Claim**: トークンや属性ストアで伝達される主体情報
- **Assurance Level (AAL/IAL)**: 本人確認・認証の保証水準
- **Identity Proofing**: 実在人物とデジタルIDを結びつける手続き
- **Identity Mapping**: 複数システム間で主体IDを対応付ける仕組み

## 7. 要点まとめ

- IAMではIdentity / Identifier / Credential / Claimを分離して設計する
- 不変ID（例: `sub`）を内部キーにし、表示用IDは属性として扱う
- 認証（本人性確認）と認可（業務許可）は同一視しない
- 日本実務では制度対応・雇用形態差分・監査証跡を前提にID対応表を整備することが重要
