# RFC 9101 — The OAuth 2.0 Authorization Framework: JWT-Secured Authorization Request (JAR)

> **発行**: IETF / 2021年8月 / **ステータス**: Proposed Standard
> **著者**: Nat Sakimura, John Bradley, Michael B. Jones / **更新**: （現時点で RFC 9101 自体を置き換える RFC はなし）

---

## 概要

RFC 9101（JAR）は、OAuth 2.0 の認可リクエストを **JWT（JWS/JWE）で保護して送るための標準仕様**である。

従来のクエリパラメータ中心の認可リクエストは、ブラウザ経由で改ざん・混在しやすいという弱点を持つ。JAR はこの課題に対し、`request`（by value）または `request_uri`（by reference）で Request Object を渡し、**完全性・送信元認証・機密性（任意）**を与える。

今日の実装では、JAR 単独よりも PAR（RFC 9126）と組み合わせる設計が主流であり、FAPI 2.0 や OpenID4VP 系プロファイルでも「署名付きリクエストを前提にする」流れが定着している。私の見立てでは、JAR は OAuth の“セキュアな入口”を定義した基礎仕様で、今後も高保証プロファイルの土台として残り続ける。

---

## 背景：なぜこの仕様が必要だったか

OAuth 2.0（RFC 6749）では、認可リクエストの多くが URL クエリに乗る。この方式は実装容易性が高い一方で、次の問題がある。

- ブラウザや中継点を通るため、パラメータ改ざん耐性が弱い
- 誰がそのリクエストを作ったかを認証しにくい
- URL 長制限や漏えいリスク（ログ・Referer）を抱える

JAR は、これを「Request Object（JWT）」に集約して解決する設計だ。言い換えると、**認可リクエストそのものを署名対象に昇格した仕様**である。

---

## 基本概念

### 1. Request Object

認可リクエストパラメータを JWT に格納したオブジェクト。`response_type`、`client_id`、`redirect_uri`、`scope`、`state`、`nonce` などを内包する。

### 2. 2つの受け渡し方法

- `request`: JWT をそのまま値として渡す（by value）
- `request_uri`: JWT を配置した URI を渡し、AS が取得する（by reference）

### 3. 保護レベル

- **JWS 署名**: 必須（改ざん検知・送信元保証）
- **JWE 暗号化**: 任意（機密性が必要な場合に追加）

---

## 主要フロー

### by value（`request`）

```
Client                         User-Agent                 Authorization Server
  |                                 |                               |
  |-- /authorize?client_id=...     |                               |
  |   &request=<signed JWT> ------>|------------------------------>|
  |                                 |                               |
  |                                 |  validate JWS/JWE,            |
  |                                 |  assemble params from JWT     |
```

### by reference（`request_uri`）

```
Client                         User-Agent                 Authorization Server
  |                                 |                               |
  |-- /authorize?client_id=...     |                               |
  |   &request_uri=https://... ---->|------------------------------>|
  |                                 |                               |
  |                                 |<------ GET request_uri -------|
  |                                 |------- signed JWT ----------->|
  |                                 |                               |
  |                                 |  validate JWS/JWE             |
```

### 実装で重要な MUST

- `request` と `request_uri` は同時指定不可
- クエリ側と Request Object 側で同一パラメータが重複していても、AS は Request Object 側を優先
- `client_id` は外側と内側で一致必須

## セキュリティ上の重要な考慮事項

RFC 9101 の Security Considerations は、実装上いまでも有効な論点が多い。

1. **アルゴリズム選択**
   弱いアルゴリズムを許可すると、JAR の価値が失われる。AS 側で許可アルゴリズムを厳格化すべき。

2. **`request_uri` の扱い**
   推測しやすい URI は情報漏えいの温床になる。十分なエントロピー・寿命制御・アクセス制御が必要。

3. **ダウングレード攻撃**
   「署名付き必須」のポリシーを AS 側で固定しないと、非保護リクエストへ誘導されうる。

4. **Cross-JWT confusion**
   ID Token 等と Request Object の JWT を混同させないため、用途ごとの検証文脈分離が必要。

私の実務感では、JAR の失敗は暗号処理そのものより **“検証ポリシーの穴”** で起きる。特に multi-tenant AS では、テナント単位で「要求強度（署名必須・PAR必須）」を明示的に設定できる設計が重要だ。

## 後継・関連仕様

| 仕様                      | 関係                     | 実務上の意味                                                             |
| ------------------------- | ------------------------ | ------------------------------------------------------------------------ |
| RFC 9126 (PAR)            | JAR と強く補完関係       | フロントチャネル露出を減らし、request_uri を AS 発行に寄せる実装が一般化 |
| RFC 9207                  | AS Issuer Identification | 認可レスポンス側の取り違え対策を強化                                     |
| RFC 9449 (DPoP)           | 送信者拘束トークン       | 認可“後”のトークン悪用耐性を強化                                         |
| FAPI 2.0 Security Profile | 高保証プロファイル       | PAR + JAR 前提で金融グレード要件を実装に落とす                           |

## 実装状況・採用

- **金融 API / Open Banking**: FAPI 系準拠で、PAR + JAR のセット採用が定番
- **OpenID4VP / OpenID4VCI 周辺**: 署名付きリクエスト（JAR 相当）を前提にした検証モデルが広がる
- **一般 OAuth サービス**: 依然としてプレーンな認可リクエスト運用は残るが、BFF 化・高リスク取引では JAR/PAR へ移行中

実装戦略としては、いきなり全クライアント強制ではなく、(1) 新規クライアント必須化 → (2) 高リスクスコープ必須化 → (3) 全体移行、の段階導入が現実的である。

---

## 読み解きのポイント

1. **JAR は「OAuth 2.0 の置き換え」ではない**
   RFC 6749 の認可要求を JWT で運ぶ拡張であり、既存概念を再利用する。

2. **JAR と PAR は役割が違う**
   JAR は“リクエストの保護”、PAR は“送信経路の最適化・保護”。併用で効果が最大化する。

3. **仕様適合と運用安全は別レイヤ**
   RFC 準拠でも、鍵ローテーション・アルゴリズム制限・`request_uri` 失効設計が弱いと事故は起きる。
