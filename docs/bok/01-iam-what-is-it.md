# 第1章 IAMとは何か

IAM（Identity and Access Management）は、**「誰が（Identity）」「何に（Resource）」「どの条件で（Policy）アクセスできるか」**を、組織として定義・運用・監査するための管理体系です。

単なるログイン機能ではなく、以下を一体で扱います。

- 本人確認と認証（Authentication）
- 権限付与と認可（Authorization）
- IDライフサイクル管理（Joiner / Mover / Leaver）
- 監査証跡と説明責任（Accountability）
- 事故時の封じ込めと復旧（Incident Response）

私の見立てでは、IAMは「セキュリティ製品の導入テーマ」ではなく、**事業継続と利用者保護を支える経営基盤**です。特に日本企業では、情シス・開発・法務・監査の分断が起きやすく、IAMを横断テーマとして扱えるかどうかが成熟度の差になります。

## 1. 概念

IAMの本質は、次の3層を同時に成立させることです。

1. **識別**: 人・デバイス・ワークロードを一意に区別する
2. **証明**: 本当にその主体かを認証で確認する
3. **統制**: 必要最小限の権限を適切な期間だけ与える

この3層のどこかが弱いと、なりすまし・権限濫用・退職者アカウント放置といったリスクが顕在化します。

## 2. なぜ必要か

IAMは「侵入を防ぐ」だけでなく、**事故を小さくする**ために必要です。

- パスワード再利用やフィッシングで認証情報が漏れる
- 過剰権限により被害範囲が拡大する
- 監査証跡が不足し、原因特定と再発防止が遅れる

加えて、クラウド/SaaS前提の業務では、アプリが増えるほどID連携と権限統制の負荷が増大します。IAMを後追いで整備すると、運用負債が雪だるま式に増えるため、初期設計での方針固定が重要です。

## 3. 代表的な構成・流れ

最小構成は次の流れで理解できます。

1. IdP（Identity Provider）にユーザーが認証
2. アプリ（SP / RP）がトークンを検証
3. アプリ内でロールや属性にもとづき認可判断
4. イベントをログに記録し、監査・検知基盤へ連携

関連仕様として、まずは以下を一次情報で押さえると体系が崩れません。

- OAuth 2.0（RFC 6749）: 認可委譲の枠組み  
  <https://datatracker.ietf.org/doc/html/rfc6749>
- OpenID Connect Core 1.0: 認証レイヤー  
  <https://openid.net/specs/openid-connect-core-1_0.html>
- NIST SP 800-63: デジタル本人確認・認証保証水準  
  <https://pages.nist.gov/800-63-4/>

## 4. よくある失敗

### 失敗1: 認証強化だけで満足する

MFAを導入しても、権限棚卸しがなければ過剰権限は残ります。認証強化と認可統制はセットで設計する必要があります。

### 失敗2: 退職・異動の権限剥奪が遅れる

Joiner/Mover/Leaverが手作業中心だと、退職者アカウント残存や権限の不整合が起きます。人事イベント連携を早期に自動化すべきです。

### 失敗3: 監査証跡を後回しにする

「ログは取っている」だけでは不十分で、誰が・いつ・どの権限で・何を実行したかを追跡できる粒度が必要です。

## 5. 日本での注意点

日本実務では、技術要件に加えて制度・運用要件を同時に満たす必要があります。

- 個人情報保護法（APPI）への対応（目的外利用の抑制、委託先管理）
- 金融・通信など業法ガイドラインへの適合
- 監査法人や内部監査に説明可能な証跡整備
- 問い合わせ窓口・アカウント回復手順など利用者保護設計

現場で特に重要なのは、**「セキュリティを上げるほど正当ユーザーが困らないか」**を定期的に評価することです。厳格な制御と業務継続性のバランスを測る運用指標（ロックアウト率、回復完了時間、権限承認リードタイムなど）を最初から置くべきです。

## 6. 最低限覚える語彙

- **Identity**: 主体を識別するための情報集合
- **Identifier**: 主体を一意に示す識別子（例: `sub`, 社員番号）
- **Credential**: 主体性を証明する手段（パスワード、鍵、証明書）
- **Authentication**: 本人性を確認するプロセス
- **Authorization**: 実行可能な操作を決めるプロセス
- **IdP**: 認証を提供する基盤
- **Least Privilege**: 必要最小限権限の原則

## 7. 要点まとめ

- IAMはログイン機能ではなく、識別・証明・統制を一体で運用する管理体系
- 認証強化だけでは不十分で、認可・ライフサイクル・監査を同時に設計する必要がある
- 日本では制度対応と利用者保護まで含めて実装・運用することが実務上の必須条件
- 導入初期から一次情報（RFC/OIDC/NIST）を参照して共通言語を揃えると、後工程の手戻りが減る
