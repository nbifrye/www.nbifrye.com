# RFC 7636 — Proof Key for Code Exchange by OAuth Public Clients

> **発行**: IETF / 2015年9月 / **ステータス**: Standards Track
> **著者**: Nat Sakimura, John Bradley, Naveen Agarwal / **更新**: OAuth 2.0 Security BCP (RFC 9700) でPKCE運用要件が強化

---

## 概要

RFC 7636（PKCE）は、OAuth 2.0 Authorization Code Flow における「認可コード奪取（authorization code interception）」を防ぐための拡張です。特に、モバイルアプリやSPAなどの公開クライアント（public client）で重要になります。クライアントは事前に `code_verifier` を生成し、そのハッシュである `code_challenge` を認可リクエストに添付します。トークン交換時に `code_verifier` の一致検証を行うことで、攻撃者がコードだけ盗んでもトークン交換できない設計です。

---

## 背景：なぜこの仕様が必要だったか

OAuth 2.0 の初期設計では、認可コード自体は「短命だから比較的安全」と見なされる場面がありました。しかし実運用では、OSのURLハンドラ競合、誤設定されたリダイレクト処理、ログ・プロキシ経由の漏えいなど、コード奪取経路が現実的に存在します。PKCEは、クライアント認証を持てない公開クライアントでも「このコードを要求した当人性」を証明できるようにした点が本質です。

私の見立てでは、PKCEは単体で万能ではありませんが、OAuthの“最低限の暗号的バインディング”として現在の実装ベースラインになっています。特に近年は、公開クライアントだけでなく confidential client でもPKCE併用が実務上ほぼ必須です。

---

## 基本概念

- **code_verifier**: クライアントが生成する高エントロピー乱数文字列
- **code_challenge**: `code_verifier` から導出した値（通常は `S256`）
- **code_challenge_method**: 変換方式。現実的には `S256` を使う
- **バインディング対象**: 発行された authorization code と `code_challenge` の対応関係

`plain` 方式も仕様上は存在しますが、現在は盗聴・ログ露出リスクを踏まえ、`S256` 前提で設計すべきです。

---

## プロトコル詳細（Authorization Code + PKCE）

```text
Client                                Authorization Server
  |--(1) code_verifier生成------------------------------->|
  |--(2) /authorize?code_challenge=...&method=S256------>|
  |<-(3) authorization code-------------------------------|
  |--(4) /token + code + code_verifier------------------->|
  |<-(5) access token / id token--------------------------|
```

### 1) 認可リクエスト

クライアントは `code_verifier` をローカル保持し、`code_challenge = BASE64URL(SHA256(code_verifier))` を送信します。

### 2) コード発行

認可サーバは authorization code と `code_challenge` を結び付けて保存します。

### 3) トークン交換

クライアントが `code_verifier` を提示し、サーバは再計算した challenge と一致するか検証します。一致しなければ拒否します。

## セキュリティ上の重要な考慮事項

1. **PKCEダウングレード対策**  
   攻撃者が `code_challenge_method` を弱い方式に落とす経路を防ぐ必要があります。サーバはPKCE処理を厳格実装し、検証失敗時は必ずエラーにすべきです。

2. **`state`/`nonce` と役割を混同しない**  
   PKCEは主にコード奪取対策です。CSRF対策（`state`）やOIDCリプレイ対策（`nonce`）とは目的が違うため、置き換えではなく併用が基本です。

3. **リダイレクトURI厳格化とセットで運用**  
   PKCEだけではオープンリダイレクトやmix-up系の問題を完全には防げません。厳格なredirect URI検証、issuer検証、PAR/JARなど上位対策と組み合わせると防御が安定します。

4. **confidential clientでもPKCE併用**  
   RFC 9700（OAuth 2.0 Security BCP）は、公開クライアントでのPKCE必須を明確化し、confidential clientでもPKCE利用を推奨しています。現場では「クライアント認証 + PKCE」を標準形にするのが妥当です。

## 実装状況・採用

現在の主要IdP/AS製品（Auth0, Okta, Entra ID, Cognito, Keycloak 等）ではPKCEが標準サポートされ、SPA/モバイルSDKもデフォルトでPKCEを使う設計が一般化しています。金融・行政分野の高保証プロファイルでも、PKCEはほぼ前提条件です。

実装レビューでは「PKCEを有効化しているか」よりも、以下を監査すると品質差が見えます。

- `S256` 強制か
- `plain` 許容時の理由と閉域条件が明文化されているか
- PKCE未指定リクエストを拒否できる設定か
- 認可コード再利用検知とログ監査があるか

---

## 読み解きのポイント

- PKCEは「公開クライアント専用機能」ではなく、今はOAuthフローの共通安全装置として扱うべき。
- PKCE導入後も、`state`・`nonce`・redirect URI検証・issuer検証は削れない。
- 仕様準拠チェックでは「パラメータの存在」だけでなく、**検証失敗時の拒否動作**まで確認することが重要。
